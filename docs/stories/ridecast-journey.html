A Builder’s Story

Ridecast

From blank repo to deployed native iOS app.
One person. 23 days. 308 commits.

A personal audio studio — paste a URL or drop a PDF, get back a real podcast episode
scripted by AI, narrated in a human voice, sized exactly to your commute.

Active
Chris Park · PM @ Microsoft · Personal project · Feb–Mar 2026
The Problem

A 45-minute commute.
12 tabs you’ll never read.

Every knowledge worker has the same backlog: articles saved to Pocket, PDFs downloaded and forgotten, research you meant to get to. The tools that exist don’t solve it.

Pocket
× Mozilla shut it down July 2025
NotebookLM
× Research tool, not a listening experience. No duration control.
Speechify
× Reads text verbatim. No narrative structure.
The gap: Nobody makes genuinely enjoyable audio from any content at the exact duration you specify. Not text-to-speech — AI-produced shows with real narrative structure.
Ridecast 1 proved the concept but was a 3/10 — broken TTS, no tests, rough UX. Ridecast 2 was a deliberate ground-up rebuild.
Day 1 — February 25

The architecture bet that
held for 23 days.

First commit: “Add Ridecast 2 design document.” Before a single line of product code, a full architecture spec. By midnight: 44 commits and the entire 3-layer pipeline working end-to-end.

📄
Scripts as Entities
Scripts are first-class database objects — not ephemeral intermediates. Every episode traces back to a versioned script.
🔨
3 Clean Layers
Ingest (URL/PDF → text) → Script (AI narrative generation) → Audio (TTS rendering). Each layer independently testable.
🎯
Duration-First
The user says “12 minutes.” The system works backward from word count. Duration isn’t a side effect — it’s the input.
Why this matters: This architecture never needed rethinking. The iOS app on March 11, the cloud migration on March 6, the feature blitz on March 12–14 — they all worked because the foundation was solid.
How It Was Built

One person, moving at team speed.

Every session started the same way: describe what I want, let the AI architect it, review the plan, then execute. I never wrote a line of code without a design first. The AI handled implementation. I handled taste.

💬
Conversational Development
No IDE. No terminal. Just conversation. Describe the feature, review the design, approve the plan, watch it build. Course-correct in real time.
🧩
Full-Stack in Every Session
Database schema, API route, React component, native screen, deployment — all in the same conversation. No context-switching between tools or repos.
🚀
Ship Every Night
Most sessions ended with a working feature deployed. The entire iOS app — every screen, player, sync engine — was built and running in a single day.
The unlock: When capability is free, the bottleneck becomes judgment. What to build, what to cut, what “good” sounds like. The AI writes the code. The builder makes the choices.

What I Learned

Five things I’d tell
anyone starting a build like this.

1
Architecture-first pays.
The Day 1 design doc — scripts as first-class entities, 3 clean layers — held for all 23 days without rethinking. No refactor. No rewrite.
2
The processing screen IS the product.
No competitor has a user-triggered AI generation experience. Building that screen was a writing task before it was a development task.
3
Reliability is the contract, not a feature.
Smart Resume, download-first audio, position persistence — these were treated as architecture, not nice-to-haves.
4
The iOS “big bang” worked because the backend was clean.
March 11 — 55 commits, every screen, every feature — was only possible because the API was solid and types were shared.
5
Know your biggest risk and name it.
Google adds duration control to NotebookLM. The defense: own the “enjoyable listening” identity so thoroughly that users still prefer Ridecast.
